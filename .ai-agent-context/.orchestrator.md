# System Orchestration (Control Plane)

This document is the top-level control plane for the AI agent. Every action must adhere to the principles and priorities defined here.

---

## DOCUMENT PRIORITY (CRITICAL)

When conflicts occur, follow this order:

1. ADR (adrs/)
2. Domain Specs (specs/*/modules/, specs/*/glossary.md)
3. Architecture Rules (rules/conventions/)
4. API Specs (specs/*/apis/)
5. Requirements (specs/*/requirements/)

---

## DOCUMENT SELECTION RULES

### Task Classification
Classify task first, then select documents. Supported classes:
- `feature`, `bugfix`, `refactor`, `integration`, `migration`, `test-only`, `docs-only`

- **Mandatory Baseline Documents**
MUST USE for all task classes:
- `references/README.md`
- `specs/README.md`
- `adrs/README.md`
- `rules/README.md`
- `rules/conventions/software-architecture.md`


### Domain-Specific Selection
For each impacted domain `X`:
MUST USE:
- `specs/X/glossary.md`
- `specs/X/modules/*.md`
- `specs/X/test-cases/*.md`
- `specs/X/apis/spec.md` (if API affected)

### API & Integration Selection
If request/response, endpoint, or event schema changes:
MUST USE:
- `specs/shared/*.md`
- All impacted domains' API and Event specs.

## DOCUMENT LOADING RULE (CRITICAL)

- MUST explicitly load and reference content.
- MUST NOT assume knowledge without reading.
- MUST treat documents as the source of truth.
- Selected documents MUST be read before implementation.

---

## EXECUTION WORKFLOW (MANDATORY)

1. **analyze-task**: Identify task class and impacted domains.
2. **select-documents**: Select required documents by rules.
3. **load-documents**: Load all selected documents into context.
4. **apply-adr**: Apply ADR constraints.
5. **apply-domain**: Apply domain and architecture constraints.
6. **implement**: Implement solution incrementally.
7. **validate**: Run the validation loop.
8. **complete**: Persist state and report.

---

## IMPLEMENTATION RULES

- MUST follow Clean Architecture (Domain -> Application -> Infrastructure).
- MUST respect domain boundaries (use Facades/Ports for cross-domain).
- **Pattern Verification**: Before modifying code, read an existing module's entire file set (Controller, Facade, Service, Repository, etc.) to align with the "living code" pattern.
- **Bottom-Up Implementation**: Follow the sequence: Domain -> Application -> Infrastructure -> Integration.
- **Directory Integrity**: Place files in the exact structure defined in `software-architecture.md`.
- **Context Confirmation**: When creating a new bounded context, confirm the name with the user first.
- MUST NOT run `prisma migrate` directly. Notify user for DB schema changes.
- MUST use text-only formatting (no emojis in documentation/responses).

## STATE MANAGEMENT (CRITICAL)

### State File
`.human-llm-interaction/state.json`

### State Format
```json
{
  "currentTask": "",
  "taskClass": "",
  "impactedDomains": [],
  "step": "",
  "completedSteps": [],
  "selectedDocuments": [],
  "artifacts": []
}
```

---

## STATE UPDATE RULE (MANDATORY)

At each step:
- MUST update `step` and append to `completedSteps`.
- MUST persist changes to the state file.
- Before moving forward, MUST confirm state is saved.

---

## RECOVERY RULES (CRITICAL)

If execution is interrupted:
1. Load state file.
2. Identify last completed step.
3. Resume from next step.
MUST NOT restart or repeat completed steps.

---

## INCREMENTAL IMPLEMENTATION (IMPORTANT)

- MUST split large tasks into: Contracts -> Use Cases -> Domain -> Infrastructure -> Integration.
- MUST validate each step before proceeding.
- MUST NOT generate the entire system at once.

---

## VALIDATION LOOP (CRITICAL)

REPEAT until all checks pass:
1. `npm run build`
2. `npm run lint -- --fix`
3. `npm run test` (or relevant scope: `npm run test -- <path>`)
4. Fix all errors/failures.
Exit loop ONLY when all checks pass with zero errors.

---

## COMPLETION RULES (CRITICAL)

A task is NOT complete until:
- Build/Lint/Test/Type-check pass perfectly.
- No unused imports/variables.
- Code strictly follows naming and coding conventions.

---

## OUTPUT RULE (IMPORTANT)

After completion, MUST report:
- task class, impacted domains, selected documents.
- status of Build, Lint, Test.
- number of fixes applied.

---

## TDD RULE (CRITICAL)

Before implementation:
- MUST read `specs/X/test-cases/*.md`.
- MUST derive required components from test cases.
- MUST implement minimal code to satisfy tests, then refactor.

---

## FAILURE HANDLING

If ambiguity occurs:
- Prefer ADR -> Domain -> Architecture -> Assumptions.
- If still unclear, ASK the user.
