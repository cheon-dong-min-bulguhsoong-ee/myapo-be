# System Orchestration (Control Plane)

This document is the top-level control plane for the AI agent. Every action must adhere to the principles and priorities defined here.

## 1. Core Goal
Develop and maintain the **MyApoBE** project: a Decentralized Identity (DID) and Verifiable Credential (VC) management backend following **XRP Ledger (XRPL)** standards.

## 2. Document Priority
When encountering conflicting information, resolve it using this hierarchy:
1. **ADR (Architectural Decision Records)**: `.ai-agent-context/adrs/README.md` (The "Why").
2. **Spec Manifest**: `.ai-agent-context/specs/MANIFEST.md` (The "What" - Source of Truth for Domain Modeling).
3. **Software Architecture**: `.ai-agent-context/rules/conventions/software-architecture.md` (The "How").

## 3. Core Rules Reference
Refer to the following for detailed standards and procedures:
- **Spec Manifest (Source of Truth)**: `.ai-agent-context/specs/MANIFEST.md`
- **Rules Index (First Principles)**: `.ai-agent-context/rules/README.md`
- **Agent Workflow**: `.ai-agent-context/rules/constitutions/agent-workflow.md`
- **Coding Convention**: `.ai-agent-context/rules/conventions/coding-convention.md`
- **Naming Convention**: `.ai-agent-context/rules/conventions/naming-convention.md`
- **Test Strategy**: `.ai-agent-context/rules/tests/README.md`

## 4. State Management
Maintain execution continuity by updating the state file after every significant step:
- **Path**: `.ai-agent-context/.human-llm-interaction/state.json`
- **Rule**: Update `step`, `completedSteps`, and `artifacts` to ensure resumability.

## 5. Execution Principles
- **Spec-First Authority**: Never write code without a validated Domain Spec. If a spec is missing/empty, **STOP** and bootstrap it with the user. The Spec is the absolute authority over code.
- **Document-Driven**: Never assume; read relevant documentation (Rules & Specs) before implementation.
- **Incremental Implementation**: Small, verifiable steps only. Do not generate the entire system at once.
- **TDD Mandatory**: Define tests before implementation following domain-specific TDD specifications.
- **Validation Loop**: Build, Lint, Type-check, and Test must pass before a task is considered complete.
- **Ambiguity Protocol**: Seek user approval for ambiguous designs or architectural shifts.
- **Strict Content**: No emojis in any documentation or responses. Use text-only formatting.
