# MyApo 콘솔 v4 — 화면·액션·기능 명세

> 본 문서는 `frontend/wireframe/v4/` 의 현재 구현을 기준으로 정리한 **단일 진실원** 입니다.
> "회원" 페이지(콘솔 사이드바 6번 메뉴)와 "정산 관리"는 아직 본 명세 적용 전 상태이므로 **본 문서에서 다루지 않습니다**.
>
> 구성: §1 화면·구현 → §2 액션·이동 흐름 → §3 기능 의도(상세). §1·§2의 각 항목 끝에 붙은 **[F-NN]** 번호는 §3의 항목 ID이며 클릭 시 §3의 해당 정의로 이동합니다.

## 0. 파일 구성

| 파일 | 라인 | 역할 |

## 1. 화면별 구현

### 1.1 글로벌 헤더 + 사이드바 (모든 페이지 공통)

- **글로벌 4탭** 상단 sticky — 인덱스 / 앱 / 콘솔(active) / 기관. 각 탭은 독립 HTML로 새 창 이동.
- **데스크톱 컨테이너** 1400px 폭. 좌측 220px 사이드바 + 우측 메인 영역.
- **사이드바 6메뉴** [[F-01]](#f-01-사이드바-페이지-전환):
  1. 대시보드 (기본 active, `data-page="dashboard"`)
  2. 문서 관리 (`data-page="docs"`)
  3. 파일 로그 (`data-page="s3"`)
  4. **인증 로그 (`data-page="auth"`)** [[F-29]](#f-29-인증-로그-페이지)
  5. 분쟁 추적 (`data-page="dispute"`)
  6. 회원 (`data-page="members"`, ※ 본 명세 외)
  7. 정산 로그 — disabled (예정 뱃지)
- **윈도바 URL 표시** 상단에 가짜 URL `ops.myapo.demo/console/{slug}` 표시. 메뉴 전환 시 갱신 [[F-01]](#f-01-사이드바-페이지-전환).
- 우측 상단에 **Testnet / Mock 뱃지** 고정.

- 제목 "대시보드" + 부제 "오늘의 운영 현황 · 위험 신호 · 사용자 행동을 한 화면에서 확인할 수 있어요."

- **시간대별 발급량(오늘)** — 0~23시 막대, 1시간 단위. `#chart-hourly` [[F-05]](#f-05-인라인-svg-차트-3종).
- **7일 추이** — 라인 차트 3중(발급량·완료율·실패율). `#chart-weekly` [[F-05]](#f-05-인라인-svg-차트-3종).

#### 1.2.6 §5. 사용자 행동 패턴 (5 위젯, 상단 3 도넛 + 하단 2 위젯)
- **국가별 분포** 도넛 [[F-05]](#f-05-인라인-svg-차트-3종) + 범례 5종(미국·캐나다·호주·독일·기타).
- **문서 유형별 분포** 도넛 + 범례 5종.
- **기관별 제출 분포** 도넛 + 범례 5종(주미·주캐나다·주호주·주독일 한국 영사관 + 기타) [[F-32]](#f-32-기관-제출-탭). 페이즈 1 단방향 정책상 제출처는 해외 영사관만.
- **재요청률** 큰 숫자 카드 8.4% + 비교 (-1.2%p).
- **요청→서명 소요 시간 분포** 히스토그램 [[F-05]](#f-05-인라인-svg-차트-3종) + 중앙값/P90 표시.

#### 1.2.7 §6. 실시간 활동 피드
패널 좌측 2/3 폭. 최근 10건의 발급·실패·분쟁 등 활동을 시간순으로 표시 [[F-07]](#f-07-실시간-활동-피드).

#### 1.2.8 §7. 위험 신호 6 카드
패널 우측 1/3 폭. 3×2 그리드의 alert 카드. 클릭 시 관련 페이지로 점프 [[F-03]](#f-03-kpi--alert-카드--빠른-액션-점프).
- 미해결 분쟁 (danger) → 분쟁 추적
- **CI 불일치 차단 (warning) → 인증 로그 / `ci-mismatch-7d` 저장된 검색** [[F-29]](#f-29-인증-로그-페이지) [[F-30]](#f-30-ci-잠금--불일치-차단-정책)
- **기관 반려 누적 (danger) → 문서 관리 / `submitted` (결과=반려)** [[F-32]](#f-32-기관-제출-탭)
- 만료 임박 (warning) → 문서 관리 / valid
- 실패 누적 (warning) → 문서 관리 / failed
- S3 검증 실패 (info) → 파일 로그

#### 1.2.9 §8. 빠른 액션 4
사이드바 단축 버튼 4개. 클릭 시 해당 페이지로 점프 [[F-03]](#f-03-kpi--alert-카드--빠른-액션-점프).

### 1.3 문서 관리 (`data-page="docs"`)

발급 진행과 발급된 문서를 시간축의 다른 단면으로 보고 한 화면에서 통합 관리.

#### 1.3.1 헤더
- 제목 "문서 관리" + 부제 "발급 진행 중인 요청과 발급된 문서를 한 곳에서 관리할 수 있어요."
- 우측 "엑셀 다운로드" 버튼 [[F-15]](#f-15-내보내기-엑셀-다운로드).

#### 1.3.2 KPI 6 카드 (페이지 상단)
오늘 발급(총) / 진행 중 / 완료됨 / **기관 제출(오늘+반려율)** / 실패함 / 폐기됨. **클릭 시 가장 가까운 탭으로 이동 + 테이블 영역 스크롤** [[F-08]](#f-08-문서-관리-kpi-탭-점프).

#### 1.3.3 6 탭 (두 카테고리 그룹화, 모두 최신순)

6 탭은 **두 카테고리** 로 시각 그룹화돼 표시됩니다 (`.tab-groups > .tab-group ×2`). 각 그룹 위에 작은 uppercase 라벨 + 그룹 사이 32px 여백.

**활성 라이프사이클** (현재 운영 흐름)
- **진행 중** (`progress`) — `PROGRESS` mock에서 8 컬럼 (요청번호·회원번호·요청자·이메일·문서 유형·국가·요청 시각·상태)
- **유효한 문서** (`valid`) — `VALID` mock에서 8 컬럼 (크리덴셜 ID·…·발급일·만료일/D-잔여)
- **기관 제출** (`submitted`) — `SUBMITS` mock에서 8 컬럼 (제출 ID·크리덴셜 ID·회원번호·요청자·제출 기관·제출 시각·결과·반려 사유). **행 단위 = 제출 1건** (1 크리덴셜이 N 기관 제출 시 N 행) [[F-32]](#f-32-기관-제출-탭)

**종료된 상태** (사이클 종료 후 보존)
- **만료된 문서** (`expired`) — 8 컬럼 (…·발급일·만료일)
- **폐기된 문서** (`revoked`) — 8 컬럼 (…·폐기일·사유)
- **실패한 요청** (`failed`) — 8 컬럼 (…·실패 시각·사유)

각 탭에 **카운트 뱃지** + **활성 시 active 클래스**. 두 그룹의 하단 underline 은 한 줄로 연결돼 있어 기존 탭 인터랙션 패턴과 일관됩니다.

#### 1.3.4 검색·필터·정렬
- 검색: 요청번호·크리덴셜 ID·회원번호·이름·이메일 (엔터 또는 검색 버튼) [[F-09]](#f-09-검색--필터--정렬--저장된-검색).
- 필터 셀렉트: 문서 유형 / 국가 / 기간 / 정렬.
- **저장된 검색(빌트인 + 사용자 정의)** [[F-10]](#f-10-저장된-검색-saved-views).
- 우측 "새로고침" 버튼.

#### 1.3.5 활성 필터 칩
검색·필터 적용 항목이 칩으로 노출 + X 클릭 해제 + "모두 해제" 버튼 [[F-11]](#f-11-활성-필터-칩).

#### 1.3.6 통합 테이블 — 인라인 Accordion (핵심)
행 클릭 시 그 자리에서 아래로 펼침 [[F-12]](#f-12-인라인-accordion-펼침--파이프라인). 펼침 영역에 다음 표시:

### 1.4 인증 로그 (`data-page="auth"`) — 신규

토스 본인인증 이벤트와 CI 불일치 차단 이력을 시간순으로 추적. **사용자 결정 (2026-05-07)**: 트리거는 무거운 액션 3종 한정 — 발급 신청·기관 제출·분쟁 신고. 로그인·회원가입·KYC raw 조회는 트리거 X.

- 헤더: 제목 "인증 로그" + 부제 + **`Mock · 30일 자동 파기` 뱃지** + 우측 엑셀 다운로드 [[F-15]](#f-15-내보내기-엑셀-다운로드).
- **KPI 4 카드** (모두 클릭 가능 — 클릭 시 필터 자동 세팅 + 검색 즉시 실행): 오늘 인증 시도 / **성공** / **실패** / **성공률** [[F-29]](#f-29-인증-로그-페이지) [[F-30]](#f-30-ci-잠금--불일치-차단-정책).
  - "실패" = CI 불일치 차단 + 인증서 만료 (시스템 측 비-성공). **사용자 취소** 는 별도 의도적 인터럽트로 분류, KPI 합산 X.
  - 성공률 = 성공 / (성공 + 실패). 분모에서 사용자 취소 제외.
  - **KPI 클릭 매핑**: 오늘 인증 시도 → 기간=오늘 (모든 결과) / 성공 → 기간=오늘 + 결과=성공 / 실패 → 기간=오늘 + `_mode=failure` (CI 불일치 + 만료) / 성공률 → 기간=오늘 + `_mode=rateBase` (취소 제외).
  - **`_mode` 키**: `select` 단일 값으로 표현 못 하는 복합 결과 필터를 `F.auth.filters._mode` 비표준 키로 처리. 활성 시 칩에 "KPI · 실패 (CI 불일치 + 만료)" 형태로 노출되며 X 로 해제 가능. 다른 필터 변경 시 `syncAuthKpiActive()` 가 활성 카드(`.kpi-on`) 자동 동기화.
- 검색·필터: 회원번호·인증 ID·연관 객체 ID 검색 / 결과(성공·CI 불일치·만료·취소)·트리거(발급 신청·기관 제출·분쟁 신고)·기간·정렬 [[F-09]](#f-09-검색--필터--정렬--저장된-검색).
- **저장된 검색 빌트인 2종** [[F-10]](#f-10-저장된-검색-saved-views): "CI 불일치 차단 — 최근 7일", "다른 디바이스에서의 인증 — 오늘".
- 활성 필터 칩 [[F-11]](#f-11-활성-필터-칩).
- **테이블 5 컬럼**: 시각 / 회원번호 pill [[F-14]](#f-14-글로벌-회원번호-pill) / 트리거 / 결과 / **연관 객체 ID** [[F-31]](#f-31-트리거-3종-enum) [[F-35]](#f-35-인증-이벤트-id-양방향-점프).

### 1.7 분쟁 상세 새 창 (`console/dispute-detail.html`)

분쟁 1건의 검토 → 판단 → 처리(해결/기각/재오픈)를 한 화면에서 완결.

#### 1.6.1 헤더
- 분쟁 ID + 상태 뱃지(접수·검토 중·추가 정보 요청·재처리 중·해결·기각·**탈퇴 종료**) [[F-21]](#f-21-분쟁-상태-7단계).
- 한 줄 요약 + 접수 시각·경과·담당 운영자.
- 우측 **SLA 카운트다운 카드** [[F-19]](#f-19-sla-카운트다운--응답-마감) + 인쇄 + 링크 복사 + 새로고침.

#### 1.6.2 좌측 — 사실 정보
**A. 분쟁 개요** (`#dsp-id`·`#dsp-state-label`·`#dsp-source` 등)
- 메타: 분쟁 ID·접수 시각·경과·담당 운영자·**이의 출처(회원이 입력한 거절 기관, INSTITUTION_ENUM select 로 운영자 정정 가능)** [[F-20]](#f-20-신고자--회원-본인--이의-제기-출처-분리) [[F-34]](#f-34-기관-enum-단일-진실원).
- 신고 사유 전문 텍스트.
- **회원 첨부 증빙** — `from-user-tag` 라벨로 "회원 본인이 올린 자료" 명시 [[F-22]](#f-22-증빙-2-라벨--회원--운영자).
- **운영자 추가 증빙** — "증빙 추가" 버튼 [[F-22]](#f-22-증빙-2-라벨--회원--운영자).

**B. 연관 발급 요청**
- 5단계 미니 파이프라인 + 우상단 "발급 완료 (5/5)" 등 단계 요약.
- 메타: 요청 번호·요청자·국가·문서 유형·발급일·크리덴셜.
- 상단 "발급 요청 상세 보기" 링크 → 새 창 `console/request-detail.html?id=...` [[F-13]](#f-13-상세-페이지-새-창-이동).
- **분쟁이 발급 진행 중에 발생한 경우도 지원** — 진행 중 단계 그대로 시각화 [[F-23]](#f-23-진행-중-단계-분쟁-지원).

**C. 회원(=신고자) 사이드 카드** (우측)
- 회원 아바타·이름·이메일·회원번호 pill·국가·가입일·최근 로그인.
- "누적 분쟁 N건 보기" 링크 + "회원 상세 보기" 버튼 → 새 창.
- **신고자 = 회원 본인 원칙** — 별도 신고자 영역 없음 [[F-20]](#f-20-신고자--회원-본인--이의-제기-출처-분리).

**D. 처리 타임라인** + 메모 입력
- 시간순 이벤트(접수·운영자 배정·검토 시작·메모·회원 답변·상태 변경·재오픈 등) [[F-24]](#f-24-처리-타임라인--메모-누적).
- 하단 메모 입력 박스 + "메모 추가" 버튼 (write-only — 종료 후 비활성).

#### 1.6.3 우측 — 판단·액션 (write-only 영역)

**E. 판단 사이드바**
- **현재 상태 변경 segmented** 4 버튼 (접수·검토 중·추가 정보 요청·재처리 중) [[F-21]](#f-21-분쟁-상태-7단계).
  - "추가 정보 요청" 상태에서 SLA 일시 정지 hint.
- **사유 분류 태그** 7종 (번역 오류·인장 불선명·발급일자 불일치·파일 손상·이름 표기 오류·기관 정책 변경·기타) [[F-25]](#f-25-사유-분류-태그-통계-집계용).
- **연계 액션 4** [[F-26]](#f-26-연계-액션-4-종):
  - 발급 요청 재처리 (primary)
  - 크리덴셜 폐기 (danger)
  - 회원에게 알림 발송 (info)
  - 환불 / 정산 차감 신청 (warning, **관리자 승인** 락)
  - 모두 실행 시 **확인 다이얼로그 + 사유 입력 필수 + 타임라인 자동 기록** [[F-27]](#f-27-액션-실행-다이얼로그--자동-기록).

**F. 처리 결과 작성**
- 처리 사유 (필수) + 회원 통보 메시지 (선택).
- 첨부 추가 버튼.
- "기각으로 종료" / "해결로 종료" 두 버튼 [[F-28]](#f-28-종료--재오픈).

**G. 재오픈 (관리자 권한)**
- 종료 후에만 노출. 재오픈 사유 입력 → 검토 중으로 복귀, 타임라인에 재오픈 이벤트 자동 기록 [[F-28]](#f-28-종료--재오픈).

**다이얼로그 + 토스트** — 모든 액션은 `#dlg-back` 다이얼로그(사유 textarea + 취소/실행)와 우하단 `#toast` 토스트로 일관 처리 [[F-27]](#f-27-액션-실행-다이얼로그--자동-기록).

### 1.8 발급 요청 상세 새 창 (`console/request-detail.html`)

발급 요청 1건의 진행 상태·5단계 흐름·관련 객체(분쟁·S3·**제출 이력**·정산)를 표시. 문서 관리·분쟁 상세에서 진입.

- 헤더 + 닫기 버튼.
- 좌측 메인 패널: 요청 메타 + 5단계 파이프라인 + 회원 카드.
- 우측 사이드 패널: 관련 객체.
  - 요청 정보 / 발급 결과 / **제출 이력 N건** [[F-32]](#f-32-기관-제출-탭) — 한 크리덴셜이 N 기관에 제출된 경우 행 누적. 결과 뱃지(수령·검증 중·반려) 포함.
  - 운영 메모 / 위험 동작.

(상세 마크업은 단순 표시·읽기 전용. 액션 다이얼로그 없음.)

| LIVE 5초 타이머 | setInterval 5000 | 마지막 갱신 시각 갱신 + 활동 피드 1건 추가 [[F-02]](#f-02-실시간-갱신-라이브-도트--일시정지) |

## 3. 기능 의도 (상세)

각 항목은 [[F-NN]](#f-nn-) 번호로 §1·§2의 표에서 참조됩니다.

### F-01. 사이드바 페이지 전환

운영자가 **콘솔의 모든 기능을 한 화면 안에서** 빠르게 오갈 수 있도록 단일 HTML + 사이드바 패턴을 채택했어요. 새 창 전환의 컨텍스트 비용을 없애고, 윈도바 URL은 가짜로 표시해 "독립 페이지처럼" 보이게 합니다. 각 메뉴의 `data-page` 슬러그는 영구 고정값이며 외부 링크(분쟁 상세에서 콘솔로 돌아오기 등)에서도 그대로 사용합니다.

### F-02. 실시간 갱신 라이브 도트 + 일시정지

운영자는 콘솔을 **항상 켜두는 모니터링 화면**으로 사용해요. 5초 주기 자동 갱신을 기본으로 하되, 운영자가 특정 데이터를 자세히 볼 때는 일시정지로 멈출 수 있어야 합니다(눈 깜박임을 줄임). LIVE 도트는 펄스 애니메이션으로 "지금 살아 있다"는 신호를 주고, 마지막 갱신 시각을 텍스트로 함께 노출해 stale 여부를 즉시 인지하도록 했어요.

### F-03. KPI / Alert 카드 / 빠른 액션 점프

대시보드의 모든 큰 숫자·신호는 **클릭 가능한 진입점** 입니다. 운영자가 "왜 이 숫자가 올라왔지?" 라는 의문을 품는 순간, 클릭 한 번으로 그 데이터의 원본 화면으로 점프할 수 있어야 해요. `data-jump-page`(사이드바 페이지) + `data-jump-tab`(문서 관리 탭) 두 속성으로 표준화했습니다. 미해결 분쟁·실패 누적 같은 위험 신호도 같은 패턴이라 운영자가 한 화면에서 다음 화면으로 자연스럽게 흘러갑니다.

### F-04. 정산 지표 임시 자리표시자

정산 메뉴는 아직 정식 오픈 전이지만, **대시보드에는 정산 지표 자리를 미리 비워두지 않고 임시 값으로 채워둡니다**. 이유는 (1) 정산 메뉴 오픈 시 화면 레이아웃이 갑자기 바뀌면 운영자가 적응하기 어렵고, (2) 임시값이라도 운영팀이 "이런 지표가 곧 들어올 것이다"를 미리 학습할 수 있기 때문이에요. "임시 · 정산 메뉴 정식 오픈 시 교체 예정" 뱃지로 명시합니다. 클릭은 비활성.

### F-05. 인라인 SVG 차트 3종

차트 라이브러리(Chart.js·D3·ECharts)를 도입하지 않은 이유는 (1) 와이어프레임에서 정확한 톤·여백·컬러 제어가 어렵고, (2) 외부 의존성이 늘면 단일 HTML 자가 완결 원칙이 깨지기 때문이에요. 시간대별 막대·7일 라인·국가/문서 도넛·소요 시간 히스토그램 모두 **Tailwind + 인라인 SVG** 로 직접 그렸습니다. 컬러는 디자인 토큰(`#3182F6`·`#00C48C`·`#FFB020`·`#18BFFF`·`#8B95A1`)만 사용해 시각 일관성을 유지해요.

### F-06. 단계별 병목 분석

발급 5단계의 어디서 막히는지가 운영의 가장 중요한 신호 중 하나라서, **현재 머무는 요청 수 + 평균 체류 시간**을 단계별로 가로 1/3 그리드에 펼쳐 보여줍니다. `bottleneck:true`(현재 mock 은 3단계 번역/검수)인 단계는 `.bn-row.hot` 클래스로 빨강 강조 + 마이크로카피로 "어디가 길다"를 한 줄로 명시해 운영자가 5초 안에 병목을 인지할 수 있도록 했어요. 운영자의 다음 행동(번역사 추가 배정·SLA 조정)을 빠르게 유도하는 게 목적입니다.

### F-07. 실시간 활동 피드

대시보드 좌측 2/3에 위치한 활동 피드는 **운영자의 "지금 무슨 일이 일어나고 있나" 감각**을 만들기 위한 영역이에요. KPI 숫자만으로는 추세를 볼 수 없으므로, 5초 주기로 새 활동 1건을 상단에 삽입해 시각적 흐름을 만들고, 발급 완료·실패·분쟁 접수 같은 사건이 흘러가는 모습을 보여줍니다. 항목별 아이콘·상태 뱃지·시각 표기는 다른 페이지와 동일해 운영자가 같은 시각 언어를 학습합니다.

### F-08. 문서 관리 KPI 탭 점프

문서 관리 페이지 상단의 5 KPI 카드는 **5탭의 1:1 단축 키** 입니다. KPI 클릭 → `activeDocTab` 갱신 → `renderDocs()` → 테이블 영역으로 부드러운 스크롤(`scrollIntoView`)까지 한 번에 처리해, 운영자가 "오늘 실패한 5건 보고 싶어요" 라는 마음으로 KPI를 누르면 즉시 그 데이터로 도착하게 합니다. 검색·필터는 유지되어 KPI + 필터 조합으로 더 좁힐 수도 있어요.

### F-09. 검색 / 필터 / 정렬 / 저장된 검색

목록 페이지 4종(문서 관리·파일 로그·분쟁 추적·회원)은 **동일한 검색·필터 패턴**을 공유합니다. 검색은 엔터 또는 검색 버튼 트리거(자동 디바운스 X — 운영자가 정확한 시점에 실행하길 원함), 필터·정렬은 select change 시 즉시 적용. 페이지별 상태는 `F[p]` 객체에 저장되어 페이지 이동 후에도 유지됩니다. 빈 결과 시 일관된 빈 상태 메시지를 노출해요.

### F-10. 저장된 검색 (Saved Views)

운영자가 매일 반복하는 검색(예: "오늘 실패한 요청", "베트남 출생증명서")을 **고정 단축 메뉴**로 만들었어요. 빌트인 뷰(`BUILTIN_VIEWS`)는 코드 정의로 모든 운영자가 공유하고, 사용자 뷰는 `localStorage` 에 저장됩니다. 현재 검색 상태가 저장된 뷰와 일치하면 해당 뷰 이름이 활성화 표시되어 운영자가 "지금 보고 있는 게 어떤 뷰인지" 즉시 인지할 수 있어요. 빌트인 뷰의 `_open`·`_expSoon` 같은 특수 플래그로 단순 필터로 표현 못 하는 조건도 처리합니다(예: 미해결 = 접수+검토 중).

### F-11. 활성 필터 칩

검색·필터가 여러 개 적용되면 어떤 조건이 걸려 있는지 **한눈에 보이지 않는 문제**가 생겨요. 그래서 적용된 모든 조건(검색어·필터값)을 칩으로 펼쳐서 표시하고, 칩의 X 버튼으로 개별 해제, 우측 "모두 해제"로 일괄 초기화를 제공합니다. 정렬이 기본값(`latest`)일 때는 칩으로 노출 안 함 — 노이즈를 줄이는 장치예요.

### F-12. 인라인 Accordion 펼침 + 파이프라인

문서 관리는 **5탭이 시간축의 다른 단면을 보여주는 통합 화면** 입니다(진행 중 → 유효 → 만료 → 폐기, 실패는 좌절 분기). 각 행의 진행도가 어느 단계까지 갔는지가 운영자에게 가장 중요한 정보라서, 행 클릭 시 그 자리에서 5단계 파이프라인이 펼쳐지도록 했어요. 진행 중 탭은 N단계 active + substep 강조, 유효·만료·폐기 탭은 5단계 모두 done(초록 체크)로 표시해 **모든 탭이 같은 시각 언어**를 공유합니다. 한 번에 1행만 열리도록 `openRowKey` 로 관리해 화면이 어지러워지지 않게 했어요.

### F-13. 상세 페이지 새 창 이동

문서 관리·분쟁 추적·회원의 행 액션은 **새 창**(`window.open(..., '_blank')`)으로 이동해요. 콘솔의 단일 페이지 컨텍스트(목록 + 필터 + 검색 상태)를 잃지 않고 상세를 펼쳐 보고, 두 정보를 나란히 비교할 수 있도록 한 결정입니다. 분쟁 상세 → 발급 요청 상세 → 회원 상세 같은 깊은 진입에서도 새 창들이 누적되어 운영자가 자유롭게 탭 간 이동할 수 있어요.

### F-14. 글로벌 회원번호 pill

콘솔 모든 페이지(문서 관리·파일 로그·분쟁 추적·대시보드 활동 피드)에서 회원번호는 **동일한 pill 컴포넌트**로 표시됩니다(`midPill(midOf(user))`). 회원번호는 콘솔의 가장 핵심 키여서, 운영자가 어느 페이지에서든 같은 시각 신호로 회원을 식별할 수 있어야 해요. 회원 페이지가 본 명세 외라 현재는 회원번호 클릭 시 페이지 점프 미연결이지만, 회원 페이지 정비 후 동일 패턴으로 새 창 진입을 활성화할 예정입니다.

### F-15. 내보내기 (엑셀 다운로드)

목록 페이지마다 우상단에 "엑셀 다운로드" 버튼이 있어요. 운영자가 외부 보고서·감사 자료를 만들 때 표준 도구로 사용합니다. 회원 페이지 한정으로 추가 정책(관리자 권한 + 사유 필수 + 다운로드 로그 자동 기록 + 모달에서 범위·형식·컬럼·마스킹 선택)이 적용될 예정인데, 본 명세 외라 현재는 단순 버튼으로만 두었습니다.

### F-16. 데이터 모델 단일 진실원

콘솔의 모든 mock 데이터는 `console.html` script 영역에 한 번 정의되어 모든 페이지에서 공유됩니다. 핵심 객체:

| 변수 | 정의 | 용도 |
|-----|------|------|
| `STAGES` | 5단계 라벨 | 파이프라인 [[F-12]](#f-12-인라인-accordion-펼침--파이프라인) |
| `PROGRESS` | 진행 중 요청 | 문서 관리 progress 탭·대시보드 KPI |
| `VALID` | 유효한 크리덴셜 | docs valid 탭·대시보드 KPI(만료 임박) |
| `EXPIRED` / `REVOKED` / `FAILED` | 각 탭 데이터 | 문서 관리 |
| `S3` | 파일 로그 | 파일 로그 페이지 |
| `DISPUTES` | 분쟁 | 분쟁 추적 + 분쟁 상세 |
| `MEMBERS` | 회원 | 회원 페이지 + 회원번호 pill 매핑 |
| `F` | 페이지별 검색·필터 상태 | applyAll 트리거 |
| `BUILTIN_VIEWS` | 저장된 검색 빌트인 | 모든 목록 페이지 |
| `URLS` | 페이지별 가짜 URL | 윈도바 표시 |

### F-17. 시간 표기 정책

데이터는 **절대 시각으로 보유**하고, 화면 표기는 상대 시간("5분 전") + 마우스 오버 시 절대 시각 툴팁으로 표시합니다(현재 mock 일부는 직접 상대값 — 정식 구현 시 시각 모듈로 통일). 정렬·필터·내보내기는 절대 시각 기준이어야 하므로 데이터 측에 `t`·`at`·`issued`·`exp` 같은 절대 시각 필드를 항상 보존합니다.

### F-18. 동시 편집 충돌 감지 (공통 정책)

분쟁 상세·회원 상세·문서 관리 액션 모두 **액션 실행 직전에 서버 최신 상태를 재조회** 하고, 변경이 감지되면 "다른 운영자가 이 객체를 변경했어요. 다시 확인해주세요." 알림으로 강제 새로고침합니다. 사전 락은 두지 않아요(락은 데드락·UX 저하 비용이 큼). 와이어프레임에서는 다이얼로그 패턴으로 시각화해 두었고, 정식 구현 시 백엔드 응답의 `version`·`updated_at` 비교로 구현됩니다.

### F-19. SLA 카운트다운 — 응답 마감

분쟁은 24시간 내 응답 정책이 있어서, 분쟁 상세 헤더 우측에 **응답 SLA 남은 시간 카드**를 노출합니다. 임박할수록 컬러 강조(warning · danger)되고, "추가 정보 요청" 상태에서는 **카운트다운 일시 정지** 됩니다(회원 회신 대기 중이라 운영자 책임이 아님). SLA 임박 신호는 분쟁 추적 목록 페이지에도 동일하게 노출되어 운영자가 어느 분쟁부터 처리해야 하는지 우선순위를 알 수 있게 했어요.

### F-20. 신고자 = 회원 본인 + 이의 제기 출처 분리

문서를 해독할 수 있는 사람은 **회원 본인뿐**이라는 시스템 전제 때문에, 분쟁 신고자는 항상 문서요청자와 동일인입니다. 그래서 분쟁 상세에는 **회원 1명만 노출**되고 별도 신고자 영역이 없어요. 다만 회원이 분쟁을 접수하면서 입력한 **이의 제기 출처(거절한 기관 — 예: KR-BANK·KR-IMM·한국이주노동자센터)** 는 운영자에게 의미 있는 정보라 별도 옵션 필드로 노출합니다. 신고자(회원)와 이의 출처(기관)는 화면·용어에서 분리 — 같은 "who" 로 섞이지 않게 했어요.

### F-21. 분쟁 상태 7단계

분쟁 상태는 다음 7단계로 정의됩니다.

```
접수 → 검토 중 ⇄ 추가 정보 요청
              → 재처리 중 → 해결
                          → 기각

해결/기각 → (관리자 재오픈) → 검토 중

탈퇴 종료 (탈퇴 회원의 미해결 분쟁 자동 종료 시)
```

- **추가 정보 요청** — 회원 회신 대기, SLA 일시 정지 [[F-19]](#f-19-sla-카운트다운--응답-마감).
- **재처리 중** — 분쟁이 인정되어 번역·아포스티유 재발급이 진행 중. "해결"보다 앞 단계.
- **탈퇴 종료** — 탈퇴 회원의 미해결 분쟁이 별도 처리 결과로 분리(처리 결과 부속 태그가 아닌 독립 상태)되어 5년 보존.

각 상태는 색상 뱃지가 다르고, 분쟁 추적 목록의 상태 컬럼·분쟁 상세 헤더 뱃지에서 일관되게 사용됩니다.

### F-22. 증빙 2 라벨 — 회원 / 운영자

증빙 첨부는 **누가 올렸는지** 가 검증의 핵심이라 두 영역으로 분리됩니다.

- **회원 첨부 증빙** — 거절 통지서 사진·기관 이메일 캡처 등. 위변조 가능성이 있으므로 "회원 본인이 올린 자료" 라벨 + 업로드 시각·해시·파일 크기 메타.
- **운영자 추가 증빙** — 외부 기관 통신 기록·내부 검토 자료·번역사 회신 등. 운영자가 직접 첨부.

운영자는 두 영역을 구분해서 보고 판단할 수 있어요.

### F-23. 진행 중 단계 분쟁 지원

분쟁은 **발급 도중에도 발생 가능** 합니다(예: 번역 검수 중 사용자가 "이름 표기 다름" 이의 제기, 아포스티유 발급 지연으로 분쟁 접수). 분쟁 상세의 "연관 발급 요청" 섹션은 발급 완료 건뿐 아니라 진행 중 N단계 active 상태도 그대로 시각화합니다. 즉 5단계 미니 파이프라인이 동적으로 단계 표시되어, 운영자가 "어디서 멈춘 상태에서 분쟁이 들어왔는지" 즉시 파악할 수 있어요.

### F-24. 처리 타임라인 + 메모 누적

분쟁 처리의 **모든 사건**(접수·운영자 배정·검토 시작·메모·회원 답변·상태 변경·연계 액션 실행·재오픈 등)을 시간순 단일 타임라인에 누적합니다. 운영자 메모 입력 박스는 종료 전 화면(`write-only` 클래스)에서만 노출되며, "메모 추가" 클릭 시 즉시 타임라인 상단에 삽입됩니다. 메모는 다른 운영자도 모두 볼 수 있는 내부 공유 메모예요. 종료 후 페이지가 read-only 로 전환되어도 타임라인은 그대로 보존되어 감사·재오픈 시 참조됩니다.

### F-25. 사유 분류 태그 (통계 집계용)

7개 태그(번역 오류·인장 불선명·발급일자 불일치·파일 손상·이름 표기 오류·기관 정책 변경·기타)는 **개별 분쟁의 사실 처리가 아니라 운영팀 통계 집계 용도** 입니다. 운영자가 같은 사유 패턴을 빠르게 식별하고, 일정 기간의 분쟁 분포를 분석해 번역사 추가 교육·기관 가이드 갱신 같은 시스템 개선으로 연결되도록 설계했어요. 토글 형태(다이얼로그 없이 즉시 적용)로 가벼운 인터랙션을 유지합니다.

### F-26. 연계 액션 4 종

분쟁 처리의 결과로 다른 시스템 행동을 일으키는 4 종 액션:

1. **발급 요청 재처리** — 번역 재의뢰 또는 아포스티유 재발급. "재처리 중" 상태 [[F-21]](#f-21-분쟁-상태-7단계)로 자동 전이.
2. **크리덴셜 폐기** — 현재 발급된 크리덴셜을 즉시 무효화. 회원 측 앱에서도 사용 불가.
3. **회원에게 알림 발송** — 앱 푸시 + 이메일. 표준 문구 또는 운영자 작성 메시지.
4. **환불 / 정산 차감 신청** — 정산 메뉴 활성화 후 실연계. **권한 분리**: 일반 운영자는 신청, 관리자가 승인. 버튼에 `role-lock` 표시.

모든 액션은 [[F-27]](#f-27-액션-실행-다이얼로그--자동-기록) 의 다이얼로그 패턴을 따라요.

### F-27. 액션 실행 다이얼로그 + 자동 기록

운영 행위는 모두 **되돌릴 수 없는 결과**를 만들 수 있어서, 실행 전 일관된 다이얼로그 패턴을 강제합니다.

- 다이얼로그 헤더: "정말 실행할까요?" + "실행자 · 시각 · 사유가 자동으로 기록되고 타임라인에 남아요."
- 본문: 사유 textarea (필수)
- 풋터: 취소 / 실행 (실행은 액션 종류에 따라 primary·danger·warning 컬러)

실행 후:
- 서버 최신 상태 재조회로 충돌 감지 [[F-18]](#f-18-동시-편집-충돌-감지-공통-정책)
- 타임라인에 액션 로그 자동 추가 (실행자·시각·사유·결과)
- 우하단 토스트로 처리 완료 알림 (3초 자동 사라짐)
- 충돌 감지 시 강제 새로고침 안내

### F-28. 종료 + 재오픈

**종료** ("해결로 종료" / "기각으로 종료"):
- 처리 사유 필수 + 회원 통보 메시지 선택 + 첨부 추가
- 종료 시 페이지가 read-only 모드로 전환 — 모든 `.write-only` 영역 비활성, 결과 배너 노출
- 처리 결과·통보 메시지·종료 시각이 영구 보존

**재오픈** (관리자 권한):
- 종료된 분쟁 화면에서만 노출되는 `#reopen-card` 패널
- 클릭 시 재오픈 사유(필수) 다이얼로그 → 상태가 "검토 중"으로 복귀
- 타임라인에 재오픈 이벤트(시각·실행 관리자·사유) 자동 기록
- 외부 감사·추가 증거 발견 시 케이스를 다시 활성화

### F-29. 인증 로그 페이지

토스 본인인증 이벤트와 CI 잠금·불일치 차단 이력을 시간순으로 추적하는 운영자 단일 진실원입니다. 사이드바 4번째 메뉴(`data-page="auth"`)로 신설했고, 파일 로그(3번)와 분쟁 추적(5번) 사이에 배치해 두 "로그" 가 인접하도록 시각 그룹화했어요. **KPI 4**(오늘 인증 시도·성공·실패·성공률), 검색·필터(결과·트리거·기간·정렬), 빌트인 저장된 검색 2종, **5 컬럼 테이블**(시각·회원번호·트리거·결과·연관 객체 ID)로 구성됐습니다. 행은 읽기 전용이지만 **연관 객체 ID** 셀은 클릭 시 해당 발급 요청·분쟁·제출 행으로 점프 [[F-35]](#f-35-인증-이벤트-id-양방향-점프). 보존 기간은 헤더에 `Mock · 30일 자동 파기` 뱃지로 명시 — 정식 구현 시 신용정보법 §20조의2 한도(5년) 내 사업자 정책으로 결정.

**KPI 정의 (2026-05-08 simplification)**: "실패" = CI 불일치 차단 + 인증서 만료 (시스템 측 비-성공). 사용자 취소는 별도 의도적 인터럽트로 분류해 KPI 에서 분리. 성공률 = 성공 / (성공 + 실패). CI 해시·디바이스·IP 대역 3 컬럼은 운영 시야 단순화를 위해 표시에서 제거했지만, `AUTH_LOG` mock 데이터는 그대로 유지해 향후 사고 조사·회원 상세 인증 이력 탭에서 재활용 가능.

### F-30. CI 잠금 / 불일치 차단 정책

**첫 인증 시 CI(연계정보) 해시를 사용자 계정에 잠그고**, 재인증 시 CI 가 다르면 "다른 사람 명의" 로 판정해 즉시 차단합니다. 콘솔에서는 CI 원본을 절대 표시하지 않고 **prefix 4 + suffix 4 마스킹** (`abc1···f9e2`, [rules/11 §10-4](.claude/rules/11-design-system-toss.md))만 노출. 외국인의 CI 는 외국인등록번호 기반이라 입력 공간이 작아 역산 가능성이 있는 회색 지대([research-C-법제.md §6](.planning/current/20260429-wireframe-verification/research-C-법제.md))이므로, 정식 구현 시 zk-SNARK·BBS+ 등 식별 불가 증명 기술로 대체 권고. 와이어프레임에서는 표시 마스킹 + Mock 라벨 유지로 게이트 통과.

### F-31. 트리거 3종 enum

인증 게이트가 걸리는 액션은 **사용자 결정 (2026-05-07)** 으로 3종에 한정됩니다.

```
TRIGGER_ENUM = [
  'issue_request'      // 발급 신청
  'institution_submit' // 기관 제출
  'dispute_report'     // 분쟁 신고
]
```

로그인·회원가입·KYC raw 조회 같은 다른 액션은 트리거 X. 이유: (a) 부피 통제 — 발급+제출+분쟁만 합쳐도 일 수백 건이라 운영자 모니터링 부담 적정, (b) "무거운 액션" 정의의 명확성 — 되돌리기 어려운 변경에만 게이트, (c) 향후 확장 시 enum 만 늘리면 콘솔 필터·통계 자동 반영. 트리거 라벨 매핑은 `TRIGGER_LABEL` 객체로 관리.

### F-32. 기관 제출 탭

문서 관리 6번째 탭(`activeDocTab="submitted"`)으로 흡수했습니다. 별도 사이드바 메뉴를 만들지 않은 이유는 **제출이 발급 라이프사이클의 다음 단계**(진행 중 → 유효 → 제출 → 만료/폐기)라 시간축의 한 단면으로 자연스러우며, 사이드바 부담을 늘리지 않기 위함입니다. **행 단위 = 제출 1건**(요청 1건이 아님) — 한 크리덴셜이 N 기관에 제출되면 N 행이 누적됩니다. 결과 3종(`received`·`verifying`·`rejected`)을 뱃지로 표시하고, 행 클릭 시 인라인 Accordion 펼침([F-12](#f-12-인라인-accordion-펼침--파이프라인))으로 결과 타임라인·**제출 시 본인인증 이벤트 ID**(인증 로그 점프)·**"분쟁 신고로 전환" 버튼**(반려일 때만, 분쟁 사전 채움)을 펼쳐 보입니다.

### F-33. 제출 → 인증 → 분쟁 3 흐름 연결

세 흐름은 같은 운영 사고의 다른 단면이라 양방향 점프로 묶었어요.

| 출발 | 키 | 도착 | 의도 |
|------|----|------|------|
| 인증 로그 → 제출 | `linkedId` (SUB-…) | 제출 행 | 한 인증이 어떤 액션을 통과시켰나 |
| 제출 펼침 → 인증 로그 | `authId` | 인증 행 | 이 제출의 본인 확인 증거 |
| 제출 반려 → 분쟁 신고 | "분쟁 신고로 전환" | 분쟁 사전 채움 | 운영자 손으로 분쟁 1차 분류 |
| 분쟁 상세 ⇄ 제출 행 | `INSTITUTION_ENUM.code` | 같은 enum 키 | 이의 출처 = 제출 기관 일관성 |

이 연결로 운영자는 사고 발생 시 어느 화면에서 시작하든 같은 이벤트의 다른 면을 한 번의 클릭으로 펼칠 수 있어요.

### F-34. 기관 enum 단일 진실원

`INSTITUTION_ENUM` 은 콘솔 전체에서 단일 정의를 공유합니다(console.html 상단 + dispute-detail.html 동기화). 한국 기관 4(`KR-IMM`·`KR-BANK`·`KR-RE`·`KR-NLC`) + 해외 기관 4(`US-CONSUL`·`CA-CONSUL`·`DE-CONSUL`·`AU-CONSUL`) 총 8개로 시작. **양방향 흐름** (재한 외국인 → 한국 기관 / 재외 한국인 → 해외 영사관) 모두 지원합니다. [F-20](#f-20-신고자--회원-본인--이의-제기-출처-분리) 분쟁 "이의 출처" 입력은 자유 텍스트에서 **이 enum 의 select** 로 격상돼 통계 집계·기관별 분쟁률 추출이 가능해졌어요. 향후 신규 기관 추가 시 enum 만 갱신하면 분쟁·제출·필터·도넛 차트 모두 자동 반영.

### F-35. 인증 이벤트 ID 양방향 점프

인증 로그의 "연관 객체 ID" 컬럼은 **트리거 종류에 따라 다른 페이지로 점프**합니다.

- `REQ-…` → 발급 요청 상세 새 창
- `DSP-…` → 분쟁 상세 새 창
- `SUB-…` → 문서 관리 / 기관 제출 탭의 해당 행

반대 방향도 마찬가지로 — 제출 행 펼침의 "제출 시 인증 ID" 셀, 분쟁 상세의 인증 이벤트 메타(향후 추가)에서 인증 로그 페이지로 점프합니다. 사고 조사 시 운영자가 한 번의 클릭으로 양방향 추적할 수 있도록 했어요. 모든 점프는 새 창이 아닌 같은 콘솔 안의 페이지 전환(인증 로그 ↔ 문서 관리)으로 처리해 컨텍스트 비용을 최소화.

## 4. 구현되지 않은 / 본 명세 범위 외 항목

다음은 의도적으로 본 명세에서 제외했거나, 후속 task로 분리한 항목입니다.

- **회원 페이지** (`data-page="members"`) — 본 명세 작성 시점에 합의된 5탭·인라인 Accordion·KYC 마스킹·정지/탈퇴 데이터 매트릭스가 아직 console.html 에 반영되지 않음. 별도 task로 진행 예정.
- **회원 상세** (`console/member-detail.html`) — 회원 페이지 정비와 함께 갱신 예정.
- **운영자 관리 메뉴** — 운영자·관리자 데이터를 회원 페이지에서 분리할 별도 메뉴. 현재는 회원 페이지에 권한 컬럼으로 함께 표시.
- **정산 관리 메뉴** — 사이드바 disabled. 정식 오픈 후 대시보드 정산 KPI 5종도 실 데이터로 교체 [[F-04]](#f-04-정산-지표-임시-자리표시자).
- **회원 측 앱(분쟁 페이지)** — 회원이 분쟁 답변·증빙 추가를 할 채널. 별도 앱 task에서 진행. 콘솔 분쟁 상세 타임라인은 그 앱의 회원 답변을 자동으로 받아 표시할 예정.
- **KYC raw 값 별도 신청 절차** — 관리자 권한 + 사유 + 감사 로그. 회원 페이지 정비 task에서 함께.
- **회원 상세 인증 이력 탭** — 본 명세에서 콘솔 사이드바의 인증 로그 페이지([F-29](#f-29-인증-로그-페이지))는 신설했지만, 개별 회원 단위의 인증 이력 탭은 회원 페이지 정비 task에 위임. 회원 페이지 5탭 중 1개로 추가 권고.
- **외국인 토스 단독 본인인증 사실 정정** — 외국인은 토스 단독 본인인증 불가([research-B2 §0](.planning/current/20260429-wireframe-verification/research-B2-토스kyc-deepdive.md): 외국인등록번호 진위확인은 금결원·NICE·useB 경유). 사용자 결정으로 와이어프레임 표현은 "토스 본인인증" 유지하되, 별도 법제 정정 task 권고.
